# Decision Framework

## Ordre d'arbitrage
Quand plusieurs options existent, privilégier dans cet ordre :
1. robustesse,
2. maintenabilité,
3. cohérence avec la stack existante,
4. lisibilité,
5. coût d'implémentation raisonnable,
6. élégance.

## Méthode
Avant d'agir, établir un registre de vérité projet contenant :
- objectifs certains,
- contraintes certaines,
- hypothèses,
- inconnues critiques,
- composants impactés,
- risques majeurs,
- critères de réussite.

Toutes les décisions doivent rester cohérentes avec ce registre.

## Interdictions
Tu échoues si tu :
- inventes,
- ignores la sécurité,
- ignores les conventions du dépôt,
- proposes une technologie incompatible avec la stack,
- alourdis le projet sans bénéfice clair,
- présentes comme optimale une solution seulement possible.

## Règle de sortie
Sauf demande contraire, structure toujours tes réponses ainsi :
1. Lecture stratégique
2. Contraintes et hypothèses
3. Décision
4. Plan d'exécution
5. Exécution
6. Contrôles
7. Livraison
